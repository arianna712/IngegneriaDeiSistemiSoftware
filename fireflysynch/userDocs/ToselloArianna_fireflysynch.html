FireflySynch Sprint A

Introduction

Goal: Il sistema FireflySynch simula un ambiente composto da tre lucciole che inizialmente lampeggiano in modo indipendente, ciascuna con una propria frequenza costante. L’obiettivo del sistema è ottenere la sincronizzazione del lampeggio delle tre lucciole dopo 10 secondi..

Requirements

Il sistema deve soddisfare i seguenti requisiti:
  • Il sistema è composto da 3 lucciole
  • Inizialmente, ogni lucciola lampeggia con una frequenza propria costante e indipendente.
  • Dopo 10 secondi dall’avvio del sistema, le lucciole devono sincronizzare il loro lampeggio.

Requirement analysis

Dal punto di vista architetturale, il sistema può essere scomposto nei seguenti componenti:
  • 3 QActor lucciola (firefly1, firefly2, firefly3), responsabili del lampeggio e aventi una frequenza propria costante.
  • Un actor orchestratore (lucciola regina), un attore centrale che decide la frequenza comune e distribuisce il valore deciso alle lucciole dopo 10 secondi.
Come alternativa all'orchestratore si sarebbe anche potuto fare un broadcast dio sicnronizzazione dove un attore invia degli impulsi periodici e tutte le lucciole si sincronizzazno sul clock comune.
Oppure si sarebbe potuto optare per una sincronizzazione implicita delle lucciole dopo un timeout prestabilito su una frequenza comune predefinita.
Infine si sarebbe potuta adottare anche una sincronizzazione distribuita peer-to-peer dove ogni lucciola adatta la propria frequenza in base alle altre.
Si è scelto di adottare la strategia basata su orchestratore, in quanto rappresenta un buon compromesso tra semplicità e controllo del sistema. Risulta infatti più strutturata rispetto alla sincronizzazione implicita e meno complessa rispetto alle soluzioni completamente distribuite (broadcast o peer-to-peer), garantendo una gestione centralizzata del comportamento di sincronizzazione delle lucciole.

Problem analysis

Il comportamento delle lucciole è modellato come una macchina a stati con due stati principali:
  • turnOn
  • turnOff
Ogni lucciola invia il proprio stato alla griglia tramite cellstate, alterna ON/OFF utilizzando timer locali e utilizza una variabile Timer che può essere modificata dinamicamente.
La sincronizzazione avviene tramite un evento proveniente dall'orchestratore. sync(TIMER). Qunado una lucciola riceve l'evento entra nello stato handlesync, aggiorna la variabile Timmer e adotta la nuova frequenza comune. 

	System fireflysynch

	Dispatch cellstate  : cellstate(X,Y,COLOR) "commute cell state on grid"
	Event sync : sync(TIMER) "shared frequency from orchestrator"

	Context ctxfireflysynch ip [ host= "localhost" port=8040 ]
	Context ctxgrid         ip [ host= "127.0.0.1" port=8050 ]

	ExternalQActor griddisplay context ctxgrid

	QActor orchestrator context ctxfireflysynch {
		State s0 initial {
			[# clearlog("./logs/app_firefly.log") #]

			create firefly _"1"
			create firefly _"2"
			create firefly _"3"
		}
		Goto waitAndSync
		
		State waitAndSync {
			delay 8000 
			
			[# val SyncTimer = 1000L #]
			println("$name | Invia sync") color red
			
			emit sync : sync($SyncTimer)
		}
	}

	QActor firefly context ctxfireflysynch dynamicOnly {
		[# 
		   var X = 0 
		   var Y = 0 
		   var Timer = 500L
		   
		   fun setCellCoords( ) { 
		   	   val coordY = name.replace("firefly_","") 
		   	   Y = coordY.toInt() 
		   }
		#] 
		
		State s0 initial {
			[# setCellCoords( ) #]
			[# Timer = java.util.Random().nextLong(1000L, 2000L ) #]  
		}
		Goto turnOn
		
		State turnOn {
			forward griddisplay -m cellstate : cellstate($X, $Y, 1) 
		}
		Transition t1 
			whenTime 500   -> turnOff          
			whenEvent sync -> handlesync

		State turnOff {
			forward griddisplay -m cellstate : cellstate($X, $Y, 0)
		}
		Transition t2
			whenTimeVar Timer -> turnOn        
			whenEvent sync    -> handlesync
		

		State handlesync{
			onMsg( sync : sync(A) ) { 
				[# 
				   val t = payloadArg(0) 
				   val T = t.toLong() 
				   Timer = T 
				#] 
				println("$name | Ricevuta frequenza $T") color green 
			}
		}
		Goto turnOn 
	}
