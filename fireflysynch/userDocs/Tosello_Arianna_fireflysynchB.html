FireflySynch – Sprint B

Introduction

Goal: Lo Sprint B estende il sistema FireflySynch realizzato nello Sprint A introducendo un sonar che rileva la presenza di ostacoli. Le tre lucciole lampeggiano inizialmente in modo indipendente; quando il sonar rileva un ostacolo a distanza inferiore a una soglia prestabilita, le lucciole devono lampeggiare in modo sincronizzato. Quando l’ostacolo si allontana oltre la soglia, le lucciole tornano a lampeggiare con frequenze casuali e indipendenti.

Requirements

Il sistema deve soddisfare i seguenti requisiti:
  • Il sistema è composto da 3 lucciole.
  • Ogni lucciola lampeggia inizialmente con una frequenza propria costante.
  • Il sistema è dotato di un sonar che misura la distanza di un ostacolo.
  • Quando la distanza rilevata è minore di una soglia DMIN, le lucciole devono lampeggiare in modo sincronizzato.
  • Quando la distanza torna maggiore o uguale a DMIN, le lucciole devono tornare a lampeggiare in modo non sincronizzato.

Requirement analysis

Dal punto di vista architetturale il sistema è composto dai seguenti componenti:
  • 3 QActor lucciola (firefly1, firefly2, firefly3), che gestiscono il lampeggio ON/OFF ed inviano il proprio stato alla griglia.
  • Un QActor orchestratore, che riceve i dati dal sonar e decide quando attivare o disattivare la sincronizzazione.
  • Un QActor sonar (sonarmock), che simula la rilevazione della distanza di un ostacolo.
L’orchestratore svolge il ruolo di componente centrale di controllo, coordinando il comportamento delle lucciole in base agli eventi provenienti dal sonar. Questa scelta permette di separare chiaramente la logica di rilevazione degli ostacoli dalla gestione del lampeggio delle lucciole. 

System fireflysynch_sonar

Dispatch cellstate : cellstate(X,Y,COLOR)
Event sonardata    : distance(D)
Event sync         : sync(TIMER)
Event unsync       : unsync(CMD)

Problem analysis

Il comportamento delle lucciole rimane modellato come una macchina a stati a due stati:
  • turnOn
  • turnOff
Ogni lucciola utilizza un timer locale per alternare lo stato ON/OFF. La durata del timer può essere modificata dinamicamente per ottenere la sincronizzazione. Il sonar emette periodicamente un evento sonardata(distance). L’orchestratore, alla ricezione di tale evento, confronta la distanza con una soglia DMIN:
  • se la distanza è minore di DMIN e il sistema non è sincronizzato, l’orchestratore invia un evento sync a tutte le lucciole;
  • se la distanza è maggiore o uguale a DMIN e il sistema risulta sincronizzato, l’orchestratore invia un evento unsync per ripristinare il comportamento casuale.

QActor orchestrator context ctxfireflysynch {
    [# 
       val DMIN = 50
       var Synced = false
    #]

    State s0 initial {
        [# clearlog("./logs/app_firefly.log") #]
        create firefly _"1"
        create firefly _"2"
        create firefly _"3"
    }
    Goto wait

    State wait { }

    Transition t0
        whenEvent sonardata -> handleSonar

    State handleSonar {
        onMsg( sonardata : distance(D) ) {
            [# val dist = payloadArg(0).toInt() #]

            if [# dist < DMIN && !Synced #] {
                [# Synced = true #]
                emit sync : sync(1000)
            }

            if [# dist >= DMIN && Synced #] {
                [# Synced = false #]
                emit unsync : unsync(ok)
            }
        }
    }
    Goto wait
}

Test plans

  • Verificare che, in assenza di ostacoli, le lucciole lampeggino con frequenze casuali.
  • Verificare che, alla rilevazione di un ostacolo sotto DMIN, le lucciole si sincronizzino.
  • Verificare che, una volta superata nuovamente DMIN, le lucciole tornino a lampeggiare in modo indipendente.

Project

Lo Sprint B riutilizza l’architettura dello Sprint A, estendendola con nuovi attori e nuovi eventi senza introdurre modifiche invasive, garantendo così modularità ed estensibilità del sistema.
